# TikTokアフィリ準備フェーズ自動化リポジトリ 仕様書

最終更新: 2026-06-08 / 対象: Claude Code + Claude in Chrome（半自動運用）

---

## 0. このリポジトリの目的

複数のTikTokアカウントを前提に、**動画作成の手前（リサーチ〜台本ドラフト）をClaude Codeに丸ごと任せる**ための作業基盤。

- アカウントごとにディレクトリを持ち、準備フェーズを独立して回す
- 結果は日付フォルダにログ化（あとで「どのリサーチがどの伸びに繋がったか」を遡れる）
- アカウント横断で**サマリ**を生成し、全体の状況を一望できる
- 人力で調べるのではなく、Claudeが収集・分析・台本下書きまで実行。あなたは出力を確認して判断するだけ

### 半自動である理由（重要）
Threadsには他人の投稿を検索収集できる公開APIが事実上ない。スクレイピングは規約違反リスクが高く採用しない。よって**Claude in Chrome があなたのログイン済みブラウザでThreadsを実際に開いて読み取る**方式を取る。完全無人ではなく「あなたがコマンドを叩けば、あとは収集→分析→台本下書きまでClaudeが進める」運用になる。

### 設計のポイント
- **アカウント追加 = `_template` をコピーして `profile.md` を書くだけ**。スキルは全アカ共通なので増設不要。これで「いずれアカウントが増える」要件を吸収する。
- ログは `accounts/<acct>/logs/<YYYY-MM-DD>/` に固定。スキルは常にこの規約でファイルを置く。
- `meta.json` に使用クエリ・件数・所要を残すことで、後から再現性を検証できる。

## 2. アカウント設定ファイル `profile.md`（雛形）

各アカウントの「前提」をここに集約。スキルはこれを読んで動く。

```markdown
# アカウント: {{handle}}

## ジャンル
転職アフィリエイト

## ターゲット
20代後半・現職に不満／自己否定感を抱えている層

## 扱う案件
- 案件名 / 報酬地点 / LP URL
- メリット（箇条書き）

## 勝ち型（このアカウントの構成テンプレ）
深い悩み指定 → 自分も同じだった → 原因をズラす → 意外な解決策 → 結果 → 軽いCTA

## Threads検索クエリ（本音が出る語に振る）
- "転職 しんどい"
- "仕事 辞めたい 理由"
- "今の自分 否定"

## 今の流行り形式（毎回更新される前提）
Vログ系 / 作業風景＋テロップ

## NG・トーン
- 煽りすぎない / 嘘の実績を作らない
```

## 3. 各スキルの仕様

### 3-0. `/discover <account>` — 工程1 候補アカウント発見

**対象: TikTok**（「制約＝型が取れていそうな人」を探す）
**手段**: サブエージェント `competitor-scanner` で TikTok のジャンル関連ハッシュタグ・キーワードを巡回
**手順**:
1. profile.md のジャンルから検索語を組み立て、TikTok内で直近伸びている投稿の発信者を収集
2. 各候補の直近20投稿を見て「フォーマットが固定化＝再現性が高いか」を判定材料として整理（○/△/×）
3. 候補アカウント・代表的な伸び投稿URL・判定理由を `00_candidates.md` に出力
4. ここで挙げた参考投稿URLは、後続の `/analyze` がそのまま入口として使う

**人の判断ポイント**: どの候補を実際に参考にするかの最終的な目利きは人が決める（スキルは候補提示まで）。

### 3-1. `/research <account>` — 工程3 Threadsリサーチ

**入力**: `accounts/<account>/profile.md` の検索クエリ群
**手段**: サブエージェント `threads-collector` を起動し、Claude in Chrome でThreadsを開く
**手順**:
1. profile.md からクエリを読む
2. 各クエリでThreads検索画面を開き、`get_page_text` で表示中の投稿本文を取得（スクロールしながら上位N件）
3. 案件・ジャンルに関係する「本音の悩み」だけをClaudeがフィルタ
4. 原文を `logs/<日付>/01_threads_raw.md` に保存
5. 表層の悩みと、その奥にある本音の2層に要約し `02_pain_points.md` に出力
6. `meta.json` に使用クエリ・取得件数・日時を記録

**人の判断ポイント**: 2層要約のうち「本音」が的を射ているかを確認・微修正。

> 規約遵守: ログイン済みセッションで「見える範囲を読む」のみ。大量自動スクロール・非公開データ取得はしない。CAPTCHA等が出たら停止して人に通知。

### 3-2. `/analyze <account>` — 工程2 競合TikTok投稿の解剖

**対象: TikTok**（Threadsではない）
**手段**: サブエージェント `competitor-scanner` で Claude in Chrome を使い **TikTok** を開く
**入口**: `00_candidates.md`（/discoverの出力）または profile.md の参考アカウント／特定の伸び投稿URL
**手順**:
1. 候補／参考TikTokアカウント・伸び投稿URLを開く
2. 冒頭フック・構成の順序・CTA・概要欄やプロフのリンク種別を分解
3. **コメント欄を読み、視聴者が刺さったポイント・頻出ワードを集計**（「こんなワードがあったから」＝消費者の言葉を拾う）
4. 複数投稿の構成を重ねて共通パターン（＝勝ち型）を抽出
5. 分解結果・頻出ワード・**参考にした投稿URL一覧**を `03_competitor.md` に出力（似た投稿のURLが手元に残る）

**人の判断ポイント**: 「なぜ伸びたか」の本質解釈、参考アカウントの最終的な目利き。

### 3-3. `/trend <account>` — 工程5 TikTokジャンル流行観測

**対象: TikTok**（ジャンル横断の広い観測。URL指定ではない）
**手段**: サブエージェント `competitor-scanner` で **TikTok** のジャンル関連ハッシュタグ・おすすめを巡回
**入口**: profile.md のジャンル（例: 転職／新卒IT）
**手順**:
1. ジャンル関連ハッシュタグ・キーワードでTikTok内を横断的に観測
2. 今このジャンルで伸びている**形式**（例: Vログ系／作業風景＋テロップ）を集計
3. 前回観測からの変化を検出し、**profile.md の「今の流行り形式」を書き換える提案**を出す

**人の判断ポイント**: profile.md を実際に更新するかは人が承認。
**推奨頻度**: 週1程度の定点観測（競合解剖が都度なのに対し、流行りはゆっくり変わるため分離している）。

### 3-4. `/offer <account>` — 工程4a 案件メリット抽出

**対象: 案件LP・条件**（悩みと繋ぐ前の素材づくり）
**手順**:
1. profile.md の案件LP URL・条件をClaudeに読ませる
2. この案件の強み・訴求点・他社との差を箇条書きで列挙 → `04a_offer_merits.md`
3. 各メリットが「どんな悩みを解きうるか」の仮説タグも付ける

**人の判断ポイント**: 列挙されたメリットの取捨選択。誇大・虚偽の訴求は除外（CLAUDE.mdのルール準拠）。

> **工程4（悩み×案件マッチング）の全体像**: `/offer` でメリットを出し（4a）、`02_pain_points.md` の悩みと総当たりでAIが接続候補を提示。**最終的な接続点（コア）は人が `04_matching.md` に確定**する。ここがあなたの勝ち筋の核なので自動化しない。

### 3-5. `/script <account>` — 工程6 台本ドラフト

**入力**: `02_pain_points.md` ＋ `04_matching.md`（悩み×案件の接続点）＋ `04a_offer_merits.md` ＋ `03_competitor.md` の勝ち型
**手順**:
1. 3ファイルを読み込む
2. ワークフローシートの台本生成プロンプト雛形に流し込む
3. フック3案・本編（勝ち型の順序）・ト書き・尺配分・維持率を取れる理由3点を出力
4. `05_script_draft.md` に保存

**人の判断ポイント**: 工程4の接続点（悩みと案件をどう繋ぐか）は価値の核なので人が確定してから実行。

### 3-6. `/summary` — 全アカ横断サマリ

**手順**:
1. `accounts/*/logs/<当日>/` を走査
2. アカウントごとに「収集件数・確定した悩み・勝ち型・台本本数・要対応」を1行に圧縮
3. `summaries/<日付>_daily.md` に表形式で出力
4. 未実行アカウントや、CAPTCHA等で止まったアカウントを警告として明示

## 4. CLAUDE.md に書く内容（憲法）

```markdown
# このリポジトリについて
TikTokアフィリの準備フェーズ（リサーチ〜台本下書き）を自動化する基盤。

## 絶対ルール
- ログは必ず accounts/<account>/logs/<YYYY-MM-DD>/ に置く
- Threadsはログイン済みChromeで「見える範囲を読む」のみ。スクレイピング・CAPTCHA回避はしない
- 投稿・送信・課金など外部に影響する操作は絶対に自動実行せず、人に確認する
- 人が握る判断（工程2の解釈・工程4の接続点・工程1の目利き）は埋めず、候補提示に留める

## 工程の定義と対象媒体
1 アカウント発見 / 2 伸びた理由の解剖【TikTok】 / 3 悩み収集【Threads】 /
4 悩み×案件マッチング / 5 流行り構成の観測【TikTok】 / 6 台本化
※悩みはThreads、競合構成・流行りはTikTok。媒体を取り違えない。

## ファイル命名規約
00_candidates / 01_threads_raw / 02_pain_points / 03_competitor / 04a_offer_merits / 04_matching / 05_script_draft / meta.json
```

## 5. 日次の回し方（運用イメージ）

```
# 1アカウントを回す
claude
> /discover yuzuha_90     # Chrome→TikTokで候補アカウント＋参考URL収集（工程1）
> （どの候補を参考にするか目利き）
> /research yuzuha_90      # Chrome→Threadsで悩み収集→2層化（悩みはThreads）
> （本音の言語化を確認・微修正）
> /offer    yuzuha_90      # 案件LPからメリット抽出（工程4a）
> /analyze  yuzuha_90      # Chrome→TikTok競合投稿を解剖・頻出ワード・参考URL記録
> （接続点 04_matching.md を自分で確定）
> /script   yuzuha_90      # 台本ドラフト生成

# 流行りは週1で十分（ジャンル全体の形式観測・TikTok）
> /trend    yuzuha_90      # profile.mdの「今の流行り形式」更新を提案

# 全アカ終わったら
> /summary                 # 当日サマリを1枚に
```

定期実行したい場合は、OSのスケジューラ（cron/タスクスケジューラ）から
`claude -p "/research yuzuha_90"` のように非対話で叩く。ただしChrome操作・人の判断が挟まる工程は半自動が前提なので、**完全無人運用はしない**。収集と下書きまでを夜間に進め、朝に判断する運用が現実的。

## 6. スコープ外（やらないこと）

- Threadsのスクレイピング、CAPTCHA回避、非公開データ取得
- TikTok等への自動投稿・自動DM（外部影響操作は人が実行）
- 報酬・課金に関わる操作の自動化
- 実績の捏造やステマ的な誘導文の生成
